What is LTS?
-----------
	Long-Term Support (LTS) versions are designed for stability, 
	providing security updates and maintenance over an extended period, 
	typically several years. 
	
	Java 21 is an LTS release, ensuring enterprises can rely on a consistent and secure environment 
	for their applications.

What is ST?
	Short-Term (ST) releases, like Java 24, focus on introducing new features and improvements quickly. 
	These versions are released every six months, catering to developers eager to experiment
	with cutting-edge enhancements before they become part of a longer-term support cycle.

	Importance of Release Model
	
	The dual release model of LTS and ST in Java provides flexibility for developers.
	It balances the need for stable, secure environments with opportunities to innovate 
	and access advanced features, encouraging both enterprise and experimental developments.

What is type inference?
-----------------------
The ability to use type inference with local variables ( var )

It reduces the verbosity of the language without compromising Java's dependable static
binding and type safety.

The compiler infers the type by using the information available in
the code, and adds it to the bytecode that it generates.

Type inference with var
-----------------------

The following lines of code show how local variable

String name = "Java Everywhere";
LocalDateTime dateTime = new LocalDateTime.now


var name = "Java Everywhere";
var dateTime = new LocalDateTime.now();


HashMap<Integer, String> map = new HashMap<Integer, String>();

var map = new HashMap<Integer, String>();
By replacing HashMap<Integer, String> with var , the preceding line of code is much
shorter

Java Sealed Classes 
--------------------

abstract class ------ sealed class ---------- final class

Introduced:
------------
	Preview in Java 15 (JEP 360)
	Finalized in Java 17 (JEP 409)

What is a Sealed Class?
----------------------
	A sealed class is a class that restricts which other 
	classes or interfaces may extend or implement it.

Why use it?
	To control inheritance
	Enforce domain modeling rules
	Improve code security and readability
	Helps compiler and tools do exhaustive checking (e.g., in switch statements)

Basic Syntax
---------------
public sealed class Shape permits Circle, Rectangle {
    // common shape logic
}

	sealed = restricted base class
	permits = list of allowed subclasses

Rules for Sealed Classes
---------------------------
	Must declare all permitted subclasses explicitly
	All permitted classes must be in the same package or module
	Each permitted subclass must be:
		final (no further extension), or
		sealed (can further restrict), or
		non-sealed (can be extended freely)


Example: Classic Shape Hierarchy
------------------------------
Base Class
-------------
public sealed class Shape permits Circle, Rectangle, Triangle {
}

Final Subclass
-----------------
public final class Circle extends Shape {
}

Sealed Subclass (restricting further)
--------------------------------------
public sealed class Rectangle extends Shape permits Square, GoldenRectangle {
}


public final class Square extends Rectangle {}

public final class GoldenRectangle extends Rectangle {}

Non-Sealed Subclass
-------------------
public non-sealed class Triangle extends Shape {
    // Anyone can extend Triangle now
}

public class Equilateral extends Triangle {}

public class Isosceles extends Triangle {}

Sealed Interfaces
-----------------------
Yes! Sealed types also work with interfaces:

public sealed interface Vehicle permits Car, Truck {}

public final class Car implements Vehicle {}
public final class Truck implements Vehicle {}
Or for functional-style enums:


public sealed interface Expr permits Constant, Add, Multiply {}

public final class Constant implements Expr {
    int value;
}

public final class Add implements Expr {
    Expr left, right;
}

public final class Multiply implements Expr {
    Expr left, right;
}


When to Use Sealed Classes
------------------------
	Use when:
		You want to model a closed hierarchy (e.g., all possible Shape types)
		You want to prevent unintended subclassing
		You want better static analysis and compiler checks

	Avoid if:
		Your design must be extensible by third parties or across modules
		You donâ€™t want to manage permitted type lists


Real-world Analogy
---------------
	Think of it like:

		sealed: You give access only to a known group
		final: No one can go further
		non-sealed: You lift all restrictions

Bonus: Migration Tip
------------------
Want to move from abstract base class to sealed?

Before:
--------
abstract class Animal {}
class Dog extends Animal {}
class Cat extends Animal {}

After:
------
public sealed class Animal permits Dog, Cat {}
public final class Dog extends Animal {}
public final class Cat extends Animal {}
